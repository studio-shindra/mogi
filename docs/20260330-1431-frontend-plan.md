### 構成の判断

- **views/ と components/ を分離**: views は router に紐づく画面単位、components は再利用可能な部品
- **api/ と mock/ を分離**: 開発時は mock/ から import し、API 完成後に api/ に切り替え。切り替えは api/*.js 内の import を変えるだけ
- **composables/**: Pinia は使わない（後述）。composables で十分
- **layout/**: ヘッダー・フッターは全画面共通なので切り出す

## 4. 状態管理の方針

### Pinia は不要（MVP）

理由:
- 画面をまたぐ共有 state がほぼない
- 予約フォームは1画面内のステップ遷移（ReserveView 内で完結）
- 受付画面も1画面内で完結
- Event / Performance データは各画面で fetch すれば十分（キャッシュ不要）

### composables で管理する state

| composable | 管理する state | 使う画面 |
|-----------|---------------|---------|
| useReservationForm | selectedPerformance, selectedTier, quantity, guestInfo, currentStep | ReserveView |
| useStaffActions | searchQuery, reservations[], loading, filters | StaffDashboardView |

### props / emit で済むもの

- PerformanceCard → EventDetailView: click イベント
- SeatTierSelector → ReserveView: tier + quantity の選択
- GuestInfoForm → ReserveView: guest 情報の入力
- StaffReservationRow → StaffDashboardView: 操作ボタンクリック

## 5. ルーティング一覧

| パス | name | view | 認証 | 備考 |
|------|------|------|------|------|
| `/` | event-list | EventListView | 不要 | 作品一覧 |
| `/:slug` | event-detail | EventDetailView | 不要 | 作品詳細 + 公演回選択 |
| `/:slug/reserve/:performanceId` | reserve | ReserveView | 不要 | 席種選択 → 情報入力 → 確定 |
| `/reservation/:token` | reservation-confirm | ReservationConfirmView | 不要 | 予約確認ページ |
| `/reservation/:token/checkin` | checkin | CheckinView | 不要 | セルフチェックイン |
| `/staff` | staff-dashboard | StaffDashboardView | **要認証** | 受付画面 |
| `/:pathMatch(.*)*` | not-found | NotFoundView | 不要 | 404 |

### ルーティング判断

- **`:slug` を使う理由**: Event は slug ベースで URL を設計済み（バックエンドと統一）
- **`/staff` を認証必須にする理由**: 受付画面は予約の修正・入場処理が可能。MVP では Django session 認証 or Basic 認証で十分
- **予約フォームを1画面にまとめる理由**: ステップごとに URL を切ると戻るボタンの state 管理が面倒。1つの ReserveView 内で currentStep を管理する方がシンプル

## 6. 画面一覧と責務

### 6.1 EventListView（作品一覧）

- **責務**: 全作品を一覧表示
- **API**: `GET /api/events/`
- **state**: events[]（fetch 結果のみ）
- **操作**: 作品クリック → EventDetailView へ遷移

### 6.2 EventDetailView（作品詳細 + 公演回選択）

- **責務**: 作品情報表示、公演回を選んで予約画面へ遷移
- **API**: `GET /api/events/:slug/`（performances ネスト済み）
- **state**: event（fetch 結果のみ）
- **操作**: 公演回クリック → ReserveView へ遷移
- **表示**: 作品名, 説明, 会場, 出演者, 公演回リスト（日時 + 残席状況）

### 6.3 ReserveView（予約フォーム — ステップ式）

- **責務**: 席種選択 → 情報入力 → 確認 → 送信
- **API**:
  - `GET /api/events/:slug/performances/:id/`（席種 + 残席）
  - `POST /api/reservations/`（予約作成）
  - `POST /api/reservations/:id/checkout/`（Stripe Checkout 開始、card の場合）
- **state**: useReservationForm composable
  - step: 1(席種) → 2(情報入力) → 3(確認) → 4(完了)
  - selectedTier, quantity, guestName, guestEmail, guestPhone, reservationType
- **card の場合**: 確定後に Stripe Checkout URL にリダイレクト
- **cash の場合**: 確定後に予約確認ページにリダイレクト

### 6.4 ReservationConfirmView（予約確認ページ）

- **責務**: 予約内容の表示、チェックインボタン表示
- **API**: `GET /api/reservations/:token/`
- **state**: reservation（fetch 結果のみ）
- **表示**: 氏名, 公演, 席種, 枚数, ステータス, 支払状態
- **操作**:
  - card/invite → 「セルフチェックイン」ボタン表示（開演1h前から）
  - cash → チェックインボタンなし（受付で処理）

### 6.5 CheckinView（セルフチェックイン）

- **責務**: チェックイン実行
- **API**: `POST /api/reservations/:token/checkin/`
- **state**: reservation, checkinResult
- **表示**: 確認画面 → チェックインボタン → 完了表示
- **制約**: card/invite のみ。cash はエラー表示。開演1h前チェック

### 6.6 StaffDashboardView（受付画面）

- **責務**: 予約検索、各種操作（現金受領、入場処理、当日券、招待確認）
- **API**:
  - `GET /api/staff/reservations/?search=&performance=`（検索）
  - `POST /api/staff/reservations/:id/mark-paid/`（現金受領）
  - `POST /api/staff/reservations/:id/check-in/`（入場処理）
  - `POST /api/staff/reservations/walk-in/`（当日券作成）
- **state**: useStaffActions composable
  - searchQuery, selectedPerformance, reservations[], loading
- **操作**:
  - 検索バー（名前 / メール / 電話 / トークン）
  - 公演回フィルタ
  - 各予約行に操作ボタン（状態に応じて出し分け）
  - 当日券登録フォーム（WalkInForm）
- **認証**: Django session 認証必須

```

Content-Type: application/json

{
  "performance_id": 1,
  "seat_tier_id": 1,
  "quantity": 2,
  "reservation_type": "card",
  "guest_name": "山田太郎",
  "guest_email": "yamada@example.com",
  "guest_phone": "090-1234-5678"
}

→ 201
{
  "id": 1,
  "token": "a1b2c3d4e5f6...",
  "performance": { "id": 1, "label": "..." },
  "seat_tier": { "id": 1, "name": "最前席", "code": "front_row" },
  "quantity": 2,
  "reservation_type": "card",
  "status": "pending",
  "payment_status": "unpaid",
  "guest_name": "山田太郎",
  "guest_email": "yamada@example.com",
  "created_at": "..."
}

→ 400 (在庫不足)
{ "detail": "残席が不足しています。最前席: 残り3枚" }

→ 400 (バリデーション)
{ "seat_tier_id": ["この席種は指定した公演に属していません"] }
```

**バリデーション**:
- seat_tier.performance == performance（整合性）
- remaining >= quantity（在庫チェック）
- invite は seat_tier.code == "rear" のみ許可
- 1 <= quantity <= 10
- card → status=pending, payment_status=unpaid
- cash → status=confirmed, payment_status=unpaid
- invite → status=confirmed, payment_status=paid

### 7.5 Stripe Checkout 開始（新規）

```
POST /api/reservations/:id/checkout/
→ 200
{
  "checkout_url": "https://checkout.stripe.com/c/pay/..."
}

→ 400
{ "detail": "この予約は Stripe 対象ではありません" }
```

- card タイプの pending 予約のみ対象
- フロントは checkout_url に `window.location.href` でリダイレクト
- Stripe 成功後、webhook が status=confirmed, payment_status=paid に更新
- Stripe 側の success_url は `/reservation/:token`（予約確認ページ）

## 9. mock 先行実装の範囲

### mock で先行実装できる画面（API 不要）

| 画面 | 理由 |
|------|------|
| EventListView | mock/events.js の配列を返すだけ |
| EventDetailView | mock/events.js の詳細データを返すだけ |
| ReserveView | ステップ遷移の UI は mock データで組める。送信は console.log で代替 |
| ReservationConfirmView | mock/reservations.js の確認データで表示 |
| CheckinView | mock/reservations.js + can_self_checkin フラグで表示 |
| StaffDashboardView | mock/reservations.js のリストで表示。操作は console.log で代替 |
| NotFoundView | 静的 |

**結論: 全画面が mock で先行実装可能。**

### mock データの切り替え方式

```javascript
// api/events.js
import { mockEvents, mockEventDetail } from '../mock/events.js'

const USE_MOCK = import.meta.env.VITE_USE_MOCK === 'true'

export async function fetchEvents() {
  if (USE_MOCK) return mockEvents
  const { data } = await client.get('/events/')
  return data
}
```

- `VITE_USE_MOCK=true` で mock モード
- API 完成後に `VITE_USE_MOCK=false` にするだけで切り替え
- mock ファイルは本番ビルドで tree-shake される（dynamic import にすればさらに確実）
