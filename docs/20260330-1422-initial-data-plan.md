#### 会場情報の置き場の判断

**Event に置く（推奨）:**
- 今回の作品は全公演同一会場
- 会場テーブルを切るほどの複雑さはない
- 将来ツアー公演（会場が変わる）なら Performance に移すか、Venue モデルを切る
- MVP では Event に `venue_name` + `venue_address` の2フィールドで十分

**Performance に置く案:**
- 公演ごとに会場が違うケースに対応できる
- ただし今回は全公演同一なので冗長

→ **Event に追加** が正解。将来 Performance に移すのは簡単。

#### 出演者情報の置き場の判断

- 今回は「コヤナギシン」のみ
- Cast テーブルを切る複雑さではない
- Event に `cast: TextField(blank)` を1フィールド追加するだけで十分
- 複数出演者は改行区切りで入れればよい（MVP）

## 3. Event モデルへの追加フィールド案

```
venue_name: CharField("会場名", max_length=200, blank=True)
venue_address: CharField("会場住所", max_length=500, blank=True)
cast: TextField("出演", blank=True)
```

- すべて blank=True（任意項目）
- 既存データに影響なし（blank なのでマイグレーションも安全）

---

## 4. 初期データ投入方法の比較

| 方法 | メリット | デメリット | 推奨度 |
|------|---------|-----------|--------|
| **admin 手入力** | 確実。GUI で確認しながら入れられる | 再現性なし。やり直しに手間 | **MVP ではこれで十分** |
| fixture (JSON/YAML) | 再投入可能。テスト用にも使える | 書くのが面倒。FK の id 依存 | △ テスト用なら後で作る |
| management command | 冪等に書ける。CI で使える | 1作品分にしては大げさ | △ 今は不要 |

**結論: admin 手入力**

理由:
- 1作品5公演のデータ量。手入力で2分
- Event 作成 → Performance をインラインで5件追加 → SeatTier は後日
- fixture は席種・価格が確定してからまとめて作る方が効率的
- admin のインラインが正しく動くか確認するテストにもなる
