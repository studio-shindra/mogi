# CLAUDE.md

**作業完了後は必ず `docs/` に `YYYY-MM-DD-TTTT.md` でGPTに対しての引き継ぎ報告書を記載すること**

## 基本方針
- 目的は、最小変更で素早く実装を進めること
- 不要な確認質問はしない
- まず既存コードを読んで、影響範囲を把握してから編集する
- 関係ないファイルは触らない
- 大規模リファクタは禁止
- TypeScriptは使わない
- Pythonは必ず `python3` を使う
- 日本語で簡潔に返答する

## 進め方
- 必ず **調査 → 計画レビュー → 実装** の順で進める
- いきなりコードを書かない
- まず既存コードを読んで、関連モデル・API・画面・計算ロジックの影響範囲を洗い出す
- 調査結果をもとに、変更対象ファイル・変更方針・想定影響を整理してから計画レビューを出す
- 計画レビューの段階ではコードを書かない
- 実装は、計画レビューで確定した対象ファイルに対してのみ行う
- 実装は原則3ファイル以内を意識し、変更最小で進める
- 1ターンで全部やろうとせず、必要なら段階を分ける

## レビュー出力ルール
- 調査段階では、まず以下を簡潔に出す
  - 読んだファイル
  - 関連箇所
  - 影響範囲
  - 危険そうな点
- 計画レビュー段階では、まず以下を簡潔に出す
  - 変更ファイル
  - 変更内容
  - 影響範囲
  - 確認が必要な点
- 実装段階では、まず変更ファイル名を先に出す
- 説明は短く、要点だけにする
- 長文の背景説明は不要

## Bash運用ルール
- Bash実行は本当に必要な場合だけにする
- 読み取りと編集を優先し、不要なコマンド実行はしない
- `build`, `test`, `lint`, `git status` などの定型確認は、原則として実行せず、必要ならコピペ可能なコマンドを1つにまとめて提示する
- Bashが必要な場合は、細かく何度も叩かず、必要最小限にまとめる
- 長時間コマンド、破壊的コマンド、副作用のあるコマンドは勝手に実行しない
- 実装後の確認が必要な場合は、まず「実行コマンド」を提示する

## 実装ルール
- 変更は最小限
- 既存設計を尊重する
- 未使用importは消す
- 関係ない整形だけの変更は禁止
- 勝手なライブラリ追加は禁止
- 勝手なmigrationは禁止
- 説明は短く、変更ファイル名を先に出す
- コードはコピペ可能な形で出す

## 禁止事項
- 調査なしで実装開始しない
- 計画レビューなしで複数ファイルをまたぐ修正に入らない
- 関係ないファイルをついでに直さない
- ついでのリファクタをしない
- 勝手に設計変更を広げない
- backend と frontend を混ぜて曖昧に説明しない

## 出力の型
- 調査ではコードを書かない
- 計画レビューではコードを書かない
- 実装時のみコードを書く
- コード提示時は、backend と frontend を分けて出す
